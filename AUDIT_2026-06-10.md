## CRITICAL â€” Fix before ANY deployment

### C1. Supabase service-role key is bundled into client JavaScript
- The key is named `VITE_SUPABASE_SERVICE_ROLE_KEY` â€” the `VITE_` prefix makes Vite
  inline it into the public browser bundle. Confirmed present in the built JS.
- Referenced client-side: `src/utils/env-check.ts:77,152`, `src/vite-env.d.ts:7`.
- Impact: any visitor can extract god-mode DB access that bypasses all RLS.
- Fix: rename to server-only `SUPABASE_SERVICE_ROLE_KEY`, remove all `src/` references,
  rebuild, grep dist for `service_role` to confirm gone, **then rotate the key**.

### C2. RLS provides no data isolation â€” policies are `USING (true)`
- App uses Clerk (not Supabase Auth) so `auth.uid()` is always NULL for the anon key.
  Migrations progressively replaced real policies with blanket `USING (true)`:
  - `transactions` SELECT+UPDATE: `20260309_*` / `20260310_fix_transactions_rls_for_anon_updates.sql`
  - `accounts`: `20260310_fix_accounts_rls_for_anon.sql`
  - `categories`: `20260311_fix_categories_rls_for_anon.sql`
  - `budgets/goals/investments/users/user_profiles`: baseline `20251030003814__initial-schema.sql:1978-2027`
  - `bank_connections` "service role" policy has `USING (true)` with NO role restriction â€”
    encrypted bank tokens readable by anon.
- "Application-level filtering by user_id" (the migrations' own comments) is not a
  security control â€” the anon key is public.
- Only DELETE was hardened (`20260320_harden_anon_delete_policies.sql`). 1 verb out of 4.
- Fix: integrate Clerk as a Supabase third-party JWT provider (so `auth.uid()`/claims
  work) and restore per-user policies, OR route all data access through Clerk-authenticated
  API handlers (the `api/data/delete-transaction.ts` pattern) and revoke anon grants.

### C3. Live secrets committed to git
- `.env.local` and `.env.test.local` are **tracked in git** with populated values:
  `CLERK_SECRET_KEY`, `VITE_SUPABASE_SERVICE_ROLE_KEY`, `ENCRYPTION_KEY`,
  `BANKING_STATE_SECRET`, `TRUELAYER_CLIENT_SECRET`.
- Fix: untrack, purge from history (`git filter-repo`), rotate every secret,
  move to Vercel env vars only.

## HIGH

### H1. The revenue path does not exist
- No Stripe webhook handler is deployed. `src/api/stripe-webhook.ts` is template code
  in `src/` (never deployed by Vercel).
- `stripeService.ts` calls `/api/subscriptions/create-checkout`, `/status`, `/api/billing/*`
  â€” none exist in `api/`. Checkout will 404 in production.
- If a customer pays, nothing updates the database. No reconciliation. Lifecycle invisible.
- Bonus: contradictory CHECK constraints on `user_profiles`
  (`initial-schema.sql:854-857`) mean the **'premium' tier cannot be written at all** â€”
  the sync trigger would fail on a premium subscription.
- Bonus 2: `stripeService.ts:196-198` defines Pro with `tier:'premium'` (duplicate);
  upgrade ladder uses tiers that don't exist.

### H2. No atomic transactions + float math on persisted balances
- Zero `BEGIN/COMMIT` anywhere. `api/data/delete-transaction.ts:53-132` performs delete â†’
  read balance â†’ write balance as 4 statements and ships a `partial_failure` error code
  for the broken state it knowingly allows.
- Float arithmetic writes to the DB: `transactionService.ts:509`, `accountService.ts:324`,
  `dataService.ts:289`, `api/data/delete-transaction.ts:118`,
  `api/banking/sync-transactions.ts:83-88`.
- Three divergent balance-update code paths that will drift; concurrent writes lose updates.
- Already cost real money once: `20260310_fix_expense_amount_signs.sql` was a data repair.
- Fix: Postgres RPCs wrapping single transactions using `SET balance = balance + $1`.

### H3. Sync layer silently loses financial data
- `autoSyncService.syncItem` (`autoSyncService.ts:510-538`) never checks the `{error}`
  return from supabase-js (which doesn't throw) â€” every offline write is marked
  'completed' even when it failed. **Guaranteed silent data loss.**
- Sync queue is in-memory only â€” page refresh discards pending ops.
- `syncService.ts` (socket.io) targets a backend that doesn't exist; drops ops after
  3 retries. Dead architecture still shipped in the bundle.
- `conflictResolutionService.ts:58,65`: merge rule 'sum' on `budget.spent` and
  `goal.currentAmount` **double-counts money** on conflict.
- `mergeData()` is a stub that discards local data (`autoSyncService.ts:369-382`).

### H4. Floating-point money math in the live frontend data path
- Docs claim "BLOCKER #1 RESOLVED â€” zero parseFloat." **False: 64 parseFloat calls in
  active src/** plus float balance updates in `AppContextSupabase.tsx:377,396,425` and
  all new-dashboard metrics (`ImprovedDashboard.tsx:62-95`).
- 22 raw float `reduce` sums on transaction amounts across components, including
  totals exported to users (`EnhancedExportManager.tsx:218-374`).
- Root cause: `Transaction.amount`/`Account.balance` typed `number` end-to-end.

### H5. Financial data persisted to plaintext localStorage
- `TransactionContext.tsx:103,107`, `GoalContext.tsx:56` write raw JSON financial
  history to localStorage, bypassing the existing `EncryptedStorageService`.
- Budgets/goals/categories never use Supabase at all in dataService
  (`dataService.ts:269-279`) despite cloud tables existing.

### H6. Pagination missing = silent data truncation
- `getTransactions` loads everything; Supabase caps at 1000 rows by default â€”
  **users with >1000 transactions silently lose data from view**
  (`transactionService.ts:126-149`).

## MEDIUM

- **Unprotected routes**: `/open-banking`, all `/settings/*` (incl. Data Management,

  (`App.tsx:253-326`). Duplicate dead route `settings/accessibility` at `App.tsx:321`.

  span+onClick, no role/tabIndex/onKeyDown (WCAG 2.1.1 failure), no error handling
  on the save path.

- **No server-side rate limiting** on any API route.

- **Raw Supabase errors returned to clients** (`delete-transaction.ts:66,109`, etc.).
- **22 `as unknown as` double-casts** in the DB mapping layer
  (`transactionService.ts` x10, `accountService.ts` x4) â€” banned by Rule #4.

- **`requireAuth` does an uncached DB round-trip** per request (`api/_lib/auth.ts:52-57`).
- **N+1 in `bulkCreateTransactions`** (~1000 round trips for a 500-row import).

- **Migration drift**: applied out-of-band via SQL editor; referenced migration missing
  from repo; 4 filename conventions; no down migrations.
